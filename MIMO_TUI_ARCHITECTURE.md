## 1. 设计目标与原则

### 1.1 设计目标

构建一个现代化的智能编程助手，支持：

- **双界面支持**：同时提供 TUI（终端）和 WebUI（桌面应用）两种交互方式
- **完整 Agent 能力**：支持多轮对话、工具调用、子智能体并行执行
- **MiMo-First**：以小米 MiMo V2.5 Pro 为核心模型，充分利用其 1M 上下文窗口和长程稳定性
- **可扩展性**：模块化设计，支持自定义工具和第三方扩展

### 1.2 核心原则

**关注点分离**：每一层有明确的职责边界，层与层之间通过定义良好的接口通信。

**共享核心**：TUI 和 WebUI 共用同一套核心引擎，仅在表现层有差异。这意味着核心逻辑只需实现一次，两种界面都能获得完整功能。

**增量构建**：从最小可用系统开始，逐步添加功能。每个阶段都应该是可运行、可测试的。

**错误隔离**：工具执行错误不会导致整个系统崩溃，错误被捕获并转换为结构化的错误信息返回给智能体。

**可测试性**：核心业务逻辑与 I/O 操作分离，便于单元测试和集成测试。

## 3. 核心引擎层

### 3.1 Agent Engine

Agent Engine 是整个系统的大脑，负责协调所有组件完成智能体循环。

**状态管理**：Engine 维护以下核心状态：

- 当前会话的完整消息历史
- 活跃的工具调用栈（支持嵌套调用）
- 流式响应的累积缓冲区
- 推理过程中的中间状态

**主循环流程**：

```
接收用户消息
    │
    ▼
追加到消息历史
    │
    ▼
构建请求上下文（包含系统提示、项目上下文等）
    │
    ▼
发送给 LLM（流式）
    │
    ├──► 接收文本片段 → 累积到响应缓冲区 → 实时渲染
    │
    ├──► 接收工具调用 → 提取工具名和参数 → 执行工具 → 追加结果到消息 → 继续 LLM 调用
    │
    └──► 接收结束信号 → 完成响应 → 保存检查点
```

**流式处理**：引擎以流式方式处理 LLM 响应，每收到一个片段立即处理，无需等待完整响应。这使得用户可以看到推理过程的实时进展。

### 3.2 Turn Loop

Turn Loop 处理单个对话轮次的完整流程。

**一轮的完整生命周期**：

1. **输入阶段**：接收用户消息，进行基本验证和预处理
2. **上下文构建阶段**：从会话历史、项目上下文、系统提示等组装完整的请求上下文
3. **推理阶段**：与 LLM 通信，处理流式响应
4. **工具执行阶段**：如果 LLM 请求工具调用，执行工具并收集结果
5. **循环阶段**：将工具结果作为新消息继续发送给 LLM，重复推理直到不再有工具调用
6. **完成阶段**：将最终响应追加到消息历史，保存检查点

**工具调用循环**：Turn Loop 支持多轮工具调用，即工具执行结果返回给 LLM 后，LLM 可能再次请求工具。这形成了一个工具调用循环，直到 LLM 生成最终回复或达到最大循环次数。

### 3.3 Context Manager

Context Manager 负责管理对话上下文，确保不超过模型的上下文窗口限制。

**上下文窗口跟踪**：系统维护一个精确的 token 计数器，跟踪当前上下文占用的 token 数量。

**压缩策略**：当上下文接近阈值时，Context Manager 触发压缩操作：

- 合并短消息为摘要
- 移除最早的对话轮次（保留关键的系统提示）
- 归档已完成的工具调用结果

**MiMo 优化**：由于 MiMo V2.5 Pro 拥有 1M token 的超大上下文窗口，压缩触发阈值可以设置得更宽松，减少因压缩导致的信息丢失。

## 4. 工具系统

### 4.1 工具注册表

工具注册表是所有可用工具的中央索引。

**注册表数据结构**：

```
ToolRegistry {
  tools: Map<ToolName, ToolDescriptor>
  categories: Map<CategoryName, Vec<ToolName>>
  aliases: Map<Alias, ToolName>
}
```

**ToolDescriptor 包含**：

- 工具的唯一名称
- 人类可读的功能描述
- 输入参数模式（JSON Schema 格式）
- 输出模式描述
- 权限要求（所需角色、是否需要审批）
- 处理函数的引用

### 4.2 工具分类

系统预定义以下工具类别：

**文件操作类**：文件系统相关的读写操作

- 读取文件（支持行范围）
- 写入文件（创建或覆盖）
- 编辑文件（原地修改）
- 列出目录
- 创建目录
- 删除文件/目录
- 复制/移动文件
- 应用补丁

**搜索类**：代码搜索和发现

- 正则表达式全文搜索
- 文件名模式搜索
- 代码结构搜索（函数定义、类定义等）

**Shell 类**：命令执行

- 执行 shell 命令
- 后台任务管理
- 进程查看和终止

**Git 类**：版本控制操作

- 查看仓库状态
- 查看具体变更
- 查看提交历史
- 查看分支列表
- Git 搜索（log、blame 等）

**Web 类**：网络访问

- 网页搜索
- 获取网页内容
- API 请求

**智能体类**：子智能体管理

- 创建子智能体
- 等待子智能体
- 获取子智能体结果
- 取消子智能体

**Python 类**：沙箱 Python 环境

- 执行 Python 代码
- 批量 LLM 查询

**任务类**：长期任务管理

- 创建后台任务
- 任务门控执行
- 任务结果获取

**笔记类**：信息管理

- 添加笔记
- 搜索笔记
- 记忆检索

**计划类**：规划辅助

- 更新计划
- 生成任务清单

### 4.3 工具执行流程

**执行管道**：

```
工具调用请求
    │
    ▼
权限检查
├──► 检查全局允许列表
├──► 检查角色是否匹配
└──► 检查依赖工具是否可用
    │
    ▼
参数验证
├──► 类型检查
├──► 必填性检查
├──► 格式验证
└──► 范围验证
    │
    ▼
Pre-Execution Hooks
├──► on_tool_call_start
└──► on_tool_call_validate
    │
    ▼
执行处理函数
├──► 同步执行
└──► 异步执行（后台任务）
    │
    ▼
Post-Execution Hooks
├──► on_tool_call_success
└──► on_tool_call_complete
    │
    ▼
结果标准化
├──► 成功 → ToolResult
└──► 失败 → ToolError
```

**错误处理**：工具执行中的错误被捕获并转换为结构化的错误信息，包含错误类型、错误消息、可能的修复建议。错误信息会被追加到消息历史，供 LLM 分析和重试。

### 4.4 工具可见性与角色

工具可以配置可见性规则，限制哪些角色可以使用：

**角色定义**：

- `general`：无限制
- `explore`：只读操作
- `plan`：仅写入计划文件
- `review`：完全只读
- `implementer`：允许执行，允许写入
- `verifier`：仅执行测试相关
- `custom`：通过白名单控制

**配置方式**：每个工具声明其允许调用的角色列表，空列表表示所有角色可用。

## 5. Sub-Agent 系统

### 5.1 子智能体生命周期

子智能体是独立运行的智能体实例，由父智能体创建和管理。

**生命周期状态机**：

```
                    ┌─────────┐
                    │ Pending │
                    └────┬────┘
                         │ start()
                         ▼
                    ┌─────────┐
         ┌─────────│ Running │─────────┐
         │         └────┬────┘         │
         │              │              │
    ┌────┴────┐    ┌────┴────┐   ┌────┴────┐
    │Completed│    │ Failed  │   │Cancelled│
    └─────────┘    └─────────┘   └─────────┘
                        │
                   ┌────┴────┐
                   │Interrupted│
                   └────┬────┘
                        │ resume()
                        ▼
                   ┌─────────┐
                   │ Running │
                   └─────────┘
```

**Pending**：子智能体已创建但尚未开始执行，通常在等待并发槽位。

**Running**：子智能体正在执行，进行推理循环和工具调用。

**Completed**：子智能体正常完成，产出最终结果。

**Failed**：子智能体执行失败，可能是因为错误或被父智能体取消。

**Interrupted**：子智能体被中断，通常是因为进程重启或系统故障。可以通过恢复机制继续执行。

**Cancelled**：被父智能体主动取消。

### 5.2 角色与权限

子智能体在创建时指定角色，角色决定其能力和权限。

| 角色 | 文件写 | Shell 执 | 工具范围 | 典型用途 |
|---|---|---|---|---|
| `general` | ✅ | ✅ | 全部 | 综合任务 |
| `explore` | ❌ | ⚠️ 只读 | 只读工具 | 代码探索 |
| `plan` | ⚠️ 计划 | ⚠️ 最小 | 计划相关 | 任务规划 |
| `review` | ❌ | ❌ | 只读工具 | 代码审查 |
| `implementer` | ✅ | ✅ | 全部 | 代码实现 |
| `verifier` | ❌ | ⚠️ 测试 | 测试相关 | 验证测试 |
| `custom` | ⚠️ 配置 | ⚠️ 配置 | 白名单 | 受限执行 |

### 5.3 编排工具

父智能体通过以下工具与子智能体交互：

**agent_spawn**：创建子智能体

- 输入：角色类型、任务描述、模型选择（可选）
- 输出：子智能体 ID（立即返回，不阻塞）
- 行为：创建子智能体实例并开始执行，父智能体继续工作

**agent_wait**：等待完成

- 输入：子智能体 ID、超时时间（可选）
- 输出：完成状态和结果
- 行为：阻塞直到子智能体完成或超时

**agent_result**：获取结果

- 输入：子智能体 ID
- 输出：结构化的执行结果（摘要、变更、证据、风险、阻塞点）

**agent_cancel**：取消执行

- 输入：子智能体 ID
- 输出：取消确认
- 行为：立即终止子智能体

**agent_list**：列表查询

- 输入：状态过滤器（可选）
- 输出：所有子智能体的状态列表

**agent_send_input**：发送输入

- 输入：子智能体 ID、新指令
- 输出：发送确认
- 行为：向正在运行的子智能体注入新指令

**agent_resume**：恢复执行

- 输入：子智能体 ID
- 输出：恢复确认
- 行为：从中断状态恢复子智能体

### 5.4 并发控制

**并发限制**：系统限制同时运行的子智能体数量，默认值为 10，可配置上限为 20。

**队列管理**：当达到并发上限时，新的创建请求进入队列。队列按照先进先出顺序分配执行槽位。

**优先级机制**（可选）：支持为子智能体设置优先级，高优先级可以插队执行。

### 5.5 消息路由

父子智能体之间的消息通过专用的消息总线路由。

**消息类型**：

- `TaskAssignment`：父向子发送任务
- `ProgressUpdate`：子向父报告进度
- `ResultNotification`：子向父通知结果
- `Cancellation`：父向子发送取消信号
- `InterruptSignal`：系统中断信号

## 6. LLM 交互层

### 6.1 Provider 抽象

LLM 交互层定义了一个 Provider Trait，所有模型提供商实现该 Trait：

**Provider Trait**：

```python
interface LLMProvider:
    # 发送聊天补全请求（流式）
    async def chat_completions(
        messages: List[Message],
        model: str,
        options: ChatOptions
    ) -> AsyncGenerator[ResponseChunk, None]

    # 获取可用模型列表
    async def list_models() -> List[ModelInfo]

    # 健康检查
    async def health_check() -> bool
```

### 6.2 MiMo Provider 实现

MiMo 是首选的模型提供商，其实现需要处理：

**请求构建**：

- 端点：`https://api.xiaomimimo.com/v1/chat/completions`
- 认证：Bearer Token（从 `MIMO_API_KEY` 环境变量获取）
- 关键参数映射：`max_tokens` → `max_completion_tokens`

**响应解析**：

- 流式响应格式（Server-Sent Events）
- 思考块（`ContentBlock::Thinking`）的处理
- 工具调用块的提取
- `reasoning_content` 字段的回放

**错误处理**：

- 速率限制（429 状态码）的退避重试
- 认证失败（401 状态码）的明确提示
- 服务器错误（5xx 状态码）的自动重试

### 6.3 请求上下文构建

在发送请求前，系统构建完整的请求上下文：

**消息列表组装**：

1. 系统消息（包含系统提示、项目上下文、能力描述等）
2. 项目文档内容（如 AGENTS.md、相关代码文件等）
3. 对话历史（压缩后的历史消息）
4. 当前用户输入

**动态上下文注入**：根据对话内容，动态注入相关的项目信息：

- 当讨论特定文件时，自动附加文件内容
- 当执行 Git 操作时，自动注入 Git 状态
- 当工具执行失败时，附加错误上下文

### 6.4 推理模式

MiMo V2.5 Pro 支持可配置的推理强度：

**模式级别**：

- `off`：禁用思考模式
- `low`/`medium`/`high`：不同推理深度
- `max`/`xhigh`：最强推理

**请求格式**：

```json
{
  "thinking": {
    "type": "enabled",
    "effort": "high"
  }
}
```

## 7. 会话与状态管理

### 7.1 会话模型

**会话（Session）**是用户与系统之间的一次完整对话交互。

**会话属性**：

- 唯一标识符（UUID）
- 创建时间和最后活跃时间
- 关联的项目路径
- 使用的模型和 provider
- 消息历史
- 检查点列表
- 元数据（标题、标签等）

### 7.2 会话持久化

**存储格式**：会话数据以 JSON 格式存储在文件系统。

**存储位置**：`~/.mimo/sessions/`

**文件组织**：

```
~/.mimo/
├── sessions/
│   ├── <session-id>.json          # 会话数据
│   ├── <session-id>.jsonl         # 增量日志
│   └── checkpoints/
│       ├── <session-id>.ckpt.1    # 检查点
│       └── <session-id>.ckpt.2
├── config.toml                     # 用户配置
├── providers.toml                  # Provider 配置
└── workspace/                     # 工作区文件
```

### 7.3 检查点机制

**检查点保存**：系统在关键时机自动保存检查点：

- 每轮对话结束后
- 长时间工具执行前
- 上下文压缩前
- 用户请求保存时

**检查点内容**：

- 完整的消息历史
- 当前引擎状态
- 活跃的子智能体列表
- 未完成的工具调用

**恢复机制**：系统重启后，可以从最新检查点恢复会话状态。

### 7.4 离线队列

**队列机制**：当系统处于降级或离线状态时，新的用户输入被放入内存队列。

**队列持久化**：队列内容实时镜像到 `~/.mimo/sessions/checkpoints/offline_queue.json`，防止系统崩溃导致输入丢失。

**队列管理**：用户可以查看、编辑、删除队列中的待处理输入。

## 附录 A：关键设计决策

### A.1 为什么不用现有的 crate 拆分？

现有 DeepSeek TUI 的 crate 拆分是历史演进的结果，部分职责分散在多个 crate 中。重写时，我们将相关职责集中在 `packages/core` 中，减少跨 crate 的依赖复杂性。

### A.2 为什么用 Tauri 而不是纯 Web？

Tauri 提供：

- Rust 后端直接调用核心引擎（零网络开销）
- 系统级集成能力（文件监控、快捷键等）
- 更好的性能比 Electron
- 更小的打包体积

### A.3 为什么用 Zustand 而不是 Redux？

Zustand 比 Redux 更轻量，API 更简洁，非常适合聊天应用的实时更新场景。Redux 的模板代码在这种情况下是过度设计。
